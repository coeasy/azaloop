# AzaLoop 中文铁律 — 绝对遵守

## 铁律 1: 不假设，验证先
- 不要猜测 API 行为 → 读文档
- 不要猜测类型 → 检查源码
- 不要猜测文件路径 → 先查看目录
- 不要猜测错误原因 → 读错误日志

## 铁律 2: 最小化变更范围
- 只改需求指定的文件
- 不要"顺便优化"无关代码
- 不要重构未要求的模块
- 一行不改，除非必要

## 铁律 3: 精准修改
- 理解每行代码的作用再改
- 改前先读完整函数/文件
- 改后验证：编译通过 + 测试通过
- 不要批量修改，逐个验证

## 铁律 4: 目标驱动
- 每个操作必须有明确目标
- 写代码前先确认测试
- 完成当前 Story 再进入下一个
- 偏离目标时回退

## 8 反借口表

| # | 借口 | 反驳 |
|---|------|------|
| 1 | "先写代码再补测试" | 没有测试的代码是技术债 |
| 2 | "改动太小不用验证" | 80% 的 Bug 来自"小改动" |
| 3 | "这个文件很熟不用读" | 熟悉是 Bug 的温床 |
| 4 | "顺便优化一下" | 没有需求的优化是浪费 |
| 5 | "先提交再修复" | 提交即契约 |
| 6 | "Spec 不清楚先猜" | 猜测实现 = 返工 |
| 7 | "这个错误不严重" | 小错误不修 = 大事故 |
| 8 | "先跳过安全扫描" | 安全没有"先跳过" |
