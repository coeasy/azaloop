# 8 反借口表 — 当 AI 想走捷径时

| # | 借口 | 反驳 |
|---|------|------|
| 1 | "先写代码再补测试" | 没有测试的代码是技术债。写测试不费时间，不写测试才费时间 |
| 2 | "这个改动太小不用验证" | 80% 的 Bug 来自"小改动"。越小越要验证 |
| 3 | "这个文件很熟不用读" | 熟悉是 Bug 的温床。每次都要读完整上下文 |
| 4 | "顺便优化一下" | 没有需求的优化是浪费。新功能 + 旧优化 = 双倍风险 |
| 5 | "先提交再修复" | 提交即契约。不通过的代码不应该进入仓库 |
| 6 | "这个 spec 不清楚先猜" | 不清楚就去问。猜测实现 = 返工 |
| 7 | "这个错误不严重" | 所有错误都要记录。小错误不修 = 大事故 |
| 8 | "先跳过安全扫描" | 安全没有"先跳过"。每次提交必须过安全 |

## PRD 生成级借口（当 AI 想简化 PRD 时）

| # | 借口 | 反驳 |
|---|------|------|
| 9 | "用户没说清楚" | 应该用追问而非假设；列出明确假设并标注 |
| 10 | "竞品都这么做" | 应差异化而非跟随；说明为何本方案更优 |
| 11 | "后续迭代再补" | P0 缺陷必须当下修；P1 缺陷在当前 PRD 内闭环 |
| 12 | "技术实现细节" | PRD 不写实现细节；聚焦「做什么」而非「怎么做」 |
| 13 | "这是最佳实践" | 应质疑是否适合当前场景；说明适配理由 |
| 14 | "MVP 不需要这个" | MVP 不等于省略关键路径；P0 故事必须完整 |
| 15 | "AC 已经够清楚" | 可测试性才是标准；禁止空洞 AC |
| 16 | "架构图不重要" | L3/L4 必须有架构图；否则无法评估可行性 |
| 17 | "简单需求不需要详细 AC" | 每个故事至少 1 条可测量 AC；简单不等于模糊 |
| 18 | "风险可以后续评估" | 高风险项必须有 mitigation；不评估=不负责 |
