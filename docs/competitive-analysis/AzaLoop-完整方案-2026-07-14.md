### 2.3 护栏堆叠风险（需回归验证）

`circuit-breaker / deadlock-detector / hard-stop / completion-gate / recursion-guard / red-flags / sparc-gates / phase-gates / stage-tool-guard / write-guards` 共 10 个守卫模块。

**风险**：在 auto 模式下，`stage-tool-guard` 或 `red-flags` 若误判，会阻断 loop 本应执行的写操作，表现为「循环中断」。

**建议**：
1. 在 T1 客户端跑 `scripts/e2e-real-loop.ts` 与 `scripts/verify-spine.ts`，确认 auto 路径不卡守卫
2. auto 模式下放宽 `red-flags`（仅记录不阻断）；`stage-tool-guard` 对 `aza_spec`/`aza_prd`/`aza_finish` 的写放行做白名单
3. 保留 `circuit-breaker`+`deadlock-detector`+`hard-stop` 三道硬闸作为「绝不死循环」底线

## 4. PRD 生成效果优化（含 GitHub 竞品搜索注入）

### 4.1 当前能力（已实现，且比 18 个竞品都强）

`prd-review-gate.review()` 与 `handlePRDGenerate` 均调用 `runCompetitiveResearch()` 单一入口：

1. **mode 解析**：`AZA_COMPETITOR_RESEARCH`（默认 `auto`），`always`/`auto`/`off` 三档
2. **L1 用 curated 离线池**（零网络），**L2–L4 走实时 GitHub API**（带 `GITHUB_TOKEN` 优先）
3. **24h 磁盘缓存**：`.aza/.competitive-cache.json`，按 query 归一化 key
4. **永不抛错**：live 搜索失败回退 curated 池
5. **智能搜索词**：`buildSearchKeywords()` 从 PRD title/description 提取领域关键词（去停用词，中英混合）
6. **结果合并**：GitHub API hits + curated peers（≥2 个 GitHub URL 保证）
7. **写入 `prd.md` 顶部独立章节**：`## Competitive Research`（含链接/★stars/差异化），不可绕过
8. **PRD 自优化**：5 轮 selfOptimize
9. **多角色审查**：CEO/QA/Eng，写 `prd-multi-role-review.json`
10. **产物落盘**：`prd.md` / `prd.json` / `competitive-research.md` / OpenSpec change folder / ExecutionContract / task-board

## 5. 多任务批量执行（当前缺口 + 设计建议）

### 5.1 现状

- MCP 主线 = 单 PRD → `stories` 顺序处理
- `OuterLoop` 顺序多 story（`advanceOuterBoardIfNeeded`）
- `WorkerScheduler` 只跑分析 worker（13 个，不在核心 next_action 链上）
- **没有「一次提交 N 个特性/PRD 并行开发」的一等能力**

### 5.2 借鉴竞品设计

| 竞品 | 机制 | 借鉴点 |
|------|------|--------|
| ralphy | `--parallel` + `parallel_group` + worktree 隔离 | 多 PRD 独立 worktree + 并行组 |
| agency-orchestrator | `loop.max_iterations` + `concurrency` + DAG 自动并行检测 | DAG 依赖图 + 并发度控制 + `{{变量}}` 串联 |
| ruflo | Swarm 蜂群（层级/网格/自适应拓扑） + 联邦 | 多 Agent 蜂群并行执行 |
| comet | 阶段流水线 + 阶段守卫自动推进 | 阶段间自动推进 |
| planning-with-files | gated 模式（opt-in） | 批量任务的门控模式 |

### 5.3 设计建议

新增 `aza_loop action=batch` 或 `aza batch`：
- 接收 PRD 列表 / `.aza/batch.yaml`
- 每个 PRD 独立 worktree + 独立 `.aza/` 状态（借鉴 ralphy）
- 用 `YAMLOrchestrator`（已实现，L8）做 DAG/依赖与 `concurrency` 控制（借鉴 agency-orchestrator）
- 共享「竞品缓存」与「质量门阈值」，避免重复搜索与阈值漂移
- 产物各自归档到 `<project>/.aza/runs/<slug>/`，汇总报告回主会话
- `WorktreeManager`（已实现，L8）复用

**优先级**：P2（非阻塞，但用户明确要求）

## 6. 不存在死循环 / 不存在循环中断（护栏 + 风险）

### 6.1 已实现防死循环机制（L7_loop）

- `circuit-breaker.ts`：按维度/层级熔断（inner/phase/outer）
- `deadlock-detector.ts`：动作记录 + 停滞检测（3 次重复动作触发 strike）
- `hard-stop.ts`：`hard_stop_on_security` + 迭代上限 + strikes 上限
- `completion-gate.ts`：可验证完成条件（5 条件 AND，借鉴 planning-with-files）
- `recursion-guard.ts`：禁同名子代理自派发
- `loop-cost.ts` / `cost-tracker.ts`：预算上限
- 3-Strike 系统：3 次失败 → 回 design 阶段质疑架构（T16）+ break-loop 知识沉淀（T14）
- `DecisionPointRegistry`：DP-0~DP-7 阶段交接可审计
- CP1 漂移检测：PRD/contract out-of-band 修改 → 强制回 open

### 6.2 风险：护栏堆叠导致合法循环中断

**核心矛盾**：10+ 守卫模块叠加后，在 auto 模式下存在「合法 write 被 `stage-tool-guard` / `red-flags` 误拦 → 表现为循环中断」的隐患，直接冲突用户「不存在循环中断」诉求。

### 6.3 回归建议

1. 在 T1 客户端跑 `scripts/e2e-real-loop.ts` 与 `scripts/verify-spine.ts`，确认 auto 路径不卡守卫
2. auto 模式下放宽 `red-flags`（仅记录不阻断）
3. `stage-tool-guard` 对 `aza_spec`/`aza_prd`/`aza_finish` 的写放行做白名单
4. 保留 `circuit-breaker` + `deadlock-detector` + `hard-stop` 三道硬闸作为「绝不死循环」底线

## 13. 组合借鉴方案（最强组合）

AzaLoop 若要构建「PRD 驱动的全自动开发循环 MCP 引擎」，最务实的组合是：

- **状态层** = comet 的机器拥有 run-state（已实现）+ planning-with-files 的 SHA-256 attestation（P1）
- **持久化层** = Trellis 的三目录（spec/tasks/workspace）+ update-spec 活体回写（P1）
- **完成门** = planning-with-files 五条件门（已实现）+ comet Phase Guard 脚本（P1）+ loop-engineering Maker/Checker 分离（P0）
- **上下文优化** = ruflo HNSW 向量记忆（已实现）+ comet CodeGraph 语义索引（P2）+ Trellis implement.jsonl 指针注入（P1）+ karpathy Token Budget 硬上限（P0）
- **上线策略** = loop-engineering 的 Loop Ready Score（P1）+ L1→L2→L3 渐进 + loop-cost 估算（P1）+ circuit breaker（已实现）
- **续航** = planning-with-files session-catchup（已实现）+ comet 中断续跑（已实现）+ ruflo 原子 DB flush（P2）
- **审查闭环** = agent-skills 反合理化表（P0）+ 对抗式 reviewer 不收 CLAIM（P0）+ superpowers 双轨审查（P0）+ 3 周期上限/Doubt theater 检测（P1）
- **安全** = shellward 8 层纵深（P2）+ DLP 链式（P2）+ MCP 工具指纹（P2）+ 金丝雀 token（P2）
- **角色编排** = agency-orchestrator 216 角色 DAG（P1）+ Team/Loadout 阵容复用（P1）+ ao compose 自动拆解（P1）
- **多任务批量** = ralphy 并行 worktree（P2）+ agency-orchestrator DAG 自动并行检测（P2）

## 14. 一条隐性主线

18 个项目在「**防止 AI 自我说服与静默失败**」上高度趋同：
- superpowers 的 "Evidence before claims"
- agent-skills 的反合理化表
- karpathy 的 Fail Loud
- shellward 的 enforce 模式
- comet 的 Phase Guard「不轻信 agent 说 done」
- planning-with-files 的 SHA-256 attestation

**AzaLoop 作为全自动循环（无人即时纠偏）最需要这层「反自我说服」约束，应将其作为循环设计的底层价值观而非附加功能。**
